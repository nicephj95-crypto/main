# Design Reference Rules

이 폴더 안의 코드는 참고용입니다.

## 기본 규칙
- design-reference 안의 코드는 실제 서비스 코드가 아닙니다.
- 실제 수정은 dispatch-frontend/src 안의 코드를 기준으로 진행합니다.
- 피그마 export 코드를 그대로 덮어쓰지 않습니다.
- 디자인만 참고하고, 기존 API 연결 / 상태관리 / 이벤트 핸들러 / 데이터 흐름은 유지합니다.
- 필요 시 className, 마크업 구조, CSS, 안전한 범위의 UI 레이아웃 정리만 진행합니다.
- 라우팅 구조와 전체 앱 구조는 바꾸지 않습니다.
- 새 라이브러리 도입이 필요하다고 판단되면 임의로 진행하지 말고 먼저 사용자에게 확인합니다.

## 작업 원칙
- 먼저 dispatch-frontend/src 안의 실제 컴포넌트를 확인합니다.
- 그 다음 design-reference 안의 피그마 코드를 참고해 UI만 반영합니다.
- 실제 동작 로직보다 디자인 참고 코드가 우선되면 안 됩니다.
- 디자인 적용 후에도 기존 기능은 동일하게 동작해야 합니다.

## 금지 사항
- 피그마 export 코드를 실제 코드에 그대로 복붙하지 않습니다.
- dispatch-frontend/src 바깥의 참고 코드를 실제 서비스 코드처럼 사용하지 않습니다.
- API 호출 방식, 상태관리 방식, 이벤트 핸들러를 임의로 변경하지 않습니다.
- 필요 이상으로 폴더 구조를 갈아엎지 않습니다.

## 미구현 기능 관련 원칙
- 피그마 디자인에 있으나 현재 실제 프로젝트에 없는 기능은 우선 UI만 구현할 수 있습니다.
- 아직 백엔드, 상태관리, 이벤트 로직이 없는 기능은 placeholder UI로만 반영합니다.
- 해당 기능은 추후 구현 예정으로 간주합니다.
- 미구현 기능 때문에 기존 동작을 깨뜨리거나 구조를 크게 변경하지 않습니다.
- 임시 버튼, 임시 입력창, 임시 섹션은 넣을 수 있지만 실제 동작은 연결하지 않습니다.
- placeholder UI는 실제 동작이 없는 상태임을 코드나 설명에서 명확히 구분합니다.

## 작업 전 확인 규칙
- 단순 스타일 수정과 안전한 마크업 정리는 바로 진행할 수 있습니다.
- 구조 변경, 상태 추가, 파일 분리, 새 기능 연결이 필요하면 먼저 사용자에게 확인합니다.
- 피그마에만 있는 기능성 UI는 placeholder로 둘지 실제 동작까지 연결할지 먼저 사용자에게 확인합니다.

## 작업 결과 보고
- 수정한 파일 목록을 먼저 정리합니다.
- 디자인만 반영한 부분과 기능까지 연결된 부분을 구분해서 설명합니다.
- placeholder로 추가한 UI가 있으면 따로 명시합니다.
- 추후 구현이 필요한 항목은 TODO 형태로 정리합니다.

## 피그마 우선 반영 원칙
- 디자인은 가능한 한 피그마를 우선 기준으로 반영합니다.
- 다만 실제 코드에 이미 존재하는 기능이 피그마 시안에 없거나 축소되어 있을 경우, 임의로 제거하거나 숨기지 않습니다.
- 피그마에 없는 기존 기능이 발견되면, 어떤 기능이 현재 존재하는지 먼저 사용자에게 정리해서 보고합니다.
- 해당 기능은 아래 중 어떤 방식으로 처리할지 사용자 확인 후 진행합니다.
  - 피그마 디자인 안에 자연스럽게 포함
  - 별도 영역으로 유지
  - 임시로 기존 UI 유지
- 기존 기능을 제거, 축소, 비노출 처리해야 하는 경우 반드시 먼저 사용자에게 확인합니다.
